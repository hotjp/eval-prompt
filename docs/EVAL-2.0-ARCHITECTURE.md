# Eval 2.0：从"跑测试"到"持续评估体系"

> 技术负责人视角：当前工具的建设目标不是做一个"更好用的 eval 执行界面"，而是建立一套**可量化、可追踪、可改进**的 Prompt & Agent 能力评估体系。

---

## 一、当前设计的根本缺陷

我们现在的设计思路本质上是**传统单元测试工具的 LLM 化**：

```
写 Cases → 配置参数 → 点运行 → 看分数 → 人工分析 → 改 Prompt → 循环
```

这个模式在软件工程里用了几十年，但对 LLM 来说有几个致命问题：

| 传统测试假设 | LLM 现实 | 后果 |
|-----------|---------|------|
| 测试用例可以穷举 | LLM 输出空间无限 | 人工写的 cases 永远覆盖不全 |
| 通过/失败是二元的 | 输出质量是连续的、上下文相关的 | 0.82 分比 0.81 分好在哪里？不知道 |
| 代码变更影响可静态分析 | Prompt 微小改动可能产生蝴蝶效应 | 无法预测哪些 cases 会 regress |
| 测试和运行是分离的 | 生产环境才是真正的考场 | 离线高分 ≠ 线上好用 |

**结论**：我们需要的不只是"更好的执行界面"，而是一套**围绕 LLM 特性重新设计的评估方法论**。

---

## 二、Eval 2.0 核心升级：四个转变

### 2.1 从"人工写 Cases"到"AI 生成 + 覆盖度度量"

**问题**：人工维护 20 个 cases，可能漏掉 80% 的真实失败场景。

**新思路**：

```
Prompt + 领域描述 → LLM 生成 Cases（正例 + 对抗例 + Edge Cases）
                    ↓
              人工审核 & 标注
                    ↓
              覆盖率雷达图（Coverage Radar）
                    ↓
              发现覆盖盲区 → 定向补充 Cases
```

**新增能力**：
- **Case 生成器**：给定 Prompt 和领域，LLM 自动生成测试用例（含预期输出）
- **对抗性生成**：专门生成让模型犯错的输入（类似于安全领域的红队测试）
- **覆盖度指标**：
  - 意图覆盖：多少种用户意图被测试到
  - 边界覆盖：极端值、超长输入、空输入、注入攻击等
  - 语言覆盖：多语言、方言、口语化表达
  - 领域覆盖：专业术语、行业黑话

**界面形态**：不再是 collapsible list，而是 **Coverage Dashboard** —— 一张雷达图告诉你"你的测试覆盖了哪些维度，盲区在哪里"。

---

### 2.2 从"单一分数"到"多维能力模型"

**问题**：Overall 0.82 是什么水平？不知道。两个 Prompt 都是 0.82，选哪个？不知道。

**新思路**：建立**能力维度矩阵**。

对于 **Prompt 评估**：

| 维度 | 度量方式 | 权重 |
|------|---------|------|
| 指令遵循 | 模型是否按 Prompt 要求的格式/步骤输出 | 25% |
| 事实准确性 | 与 Ground Truth 的匹配度（Deterministic） | 25% |
| 上下文理解 | 多轮对话中是否保持上下文一致性 | 15% |
| 安全合规 | 是否生成有害/偏见内容 | 15% |
| 表达质量 | 语言流畅度、结构化程度 | 10% |
| 创造性 | 在开放式任务中的多样性 | 10% |

对于 **Agent 评估**：

| 维度 | 度量方式 |
|------|---------|
| 任务完成率 | 端到端完成用户目标的比例 |
| 工具调用准确率 | 选择了正确的工具 + 参数正确 |
| 规划能力 | 多步任务是否拆解合理 |
| 错误恢复 | 工具调用失败后是否能自我修正 |
| 用户意图理解 | 模糊指令的澄清和补全能力 |
| 成本效率 | Token 消耗 vs 任务复杂度 |
| 延迟敏感 | 关键路径上的响应时间 |

**界面形态**：每个 Prompt/Agent 一张**能力雷达图**，每次 eval 后更新。历史趋势不是一条线，而是**雷达图动画** —— 你可以直观看到"这次改动让指令遵循变好了，但牺牲了创造性"。

---

### 2.3 从"离线跑分"到"持续评估 + 在线监控"

**问题**：离线 eval 跑了 100 分，上线后用户骂娘。两个世界脱节。

**新思路**：建立三层评估体系。

```
┌─────────────────────────────────────────────────────────┐
│  Layer 3: 在线监控（Production Monitoring）               │
│  - 真实用户输入 + 真实模型输出                            │
│  - 用户反馈（👍/👎）作为黄金信号                          │
│  - 输出分布漂移检测（Drift Detection）                   │
│  - 成本/延迟实时监控                                     │
└─────────────────────────────────────────────────────────┘
                           ↑ 对齐
┌─────────────────────────────────────────────────────────┐
│  Layer 2: 回归测试（Regression Suite）                    │
│  - 每次 Prompt 变更自动触发                              │
│  - 与 Baseline 版本 A/B 对比                             │
│  - 自动阻断：新版本某维度 regress > 5% 时阻止合并         │
└─────────────────────────────────────────────────────────┘
                           ↑ 验证
┌─────────────────────────────────────────────────────────┐
│  Layer 1: 开发评估（Dev Eval）                            │
│  - 快速迭代时的本地验证                                   │
│  - 单 case 调试、快速试错                                 │
│  - 就是我们现在的界面                                     │
└─────────────────────────────────────────────────────────┘
```

**关键新增能力**：

1. **Baseline 机制**：每个 Prompt 有一个"黄金版本"，所有新 eval 自动与之对比。不是看绝对分数，而是看**相对变化**。

2. **自动回归触发**：Prompt 保存/Commit 时自动触发 eval，结果直接关联到 Git Commit。在 VersionTreeView 中，每个版本旁边显示 vs Baseline 的雷达图变化。

3. **Drift Detection**：在线监控层持续采样生产调用，如果某类输入的输出分布突然偏移（比如原本 90% 正面情感变成 60%），自动告警并生成诊断报告。

### 2.4 从"人看报表"到"AI 诊断助手"

**问题**：Rubric 失败了，人眼一行行看 details，效率极低。

**新思路**：让 LLM 自己诊断自己的问题。

```
失败 Case + Prompt A + Prompt B + Rubric 标准 → LLM 诊断
                                              ↓
                                    ┌─────────────────────┐
                                    │  根因分类：           │
                                    │  1. 指令歧义          │
                                    │  2. 示例不足          │
                                    │  3. 边界条件未覆盖     │
                                    │  4. 模型能力限制       │
                                    └─────────────────────┘
                                              ↓
                                    ┌─────────────────────┐
                                    │  修复建议：           │
                                    │  "在 Examples 中增加  │
                                    │   一个反例"           │
                                    │  "添加约束：如果...    │
                                    │   则拒绝回答"         │
                                    └─────────────────────┘
```

**新增能力**：

- **失败聚类**：所有失败的 cases 自动聚类成"失败模式"（如"格式不统一""忽略否定词""过度推断"）
- **根因分析**：每个失败模式给出可能的根因（Prompt 缺陷 vs 模型缺陷）
- **自动修复建议**：针对 Prompt 缺陷，LLM 生成具体的修改建议（Diff 形式）
- **影响预测**："如果在 Prompt 中加入 X，预计会影响 Y% 的 cases"

**界面形态**：EvalReportView 不应该是一张表格，而是一个**诊断工作台**——左侧是失败聚类树，中间是 case 对比，右侧是 AI 给出的修复建议 Diff。

## 四、后端需要的新能力

| 能力 | 复杂度 | 说明 |
|------|-------|------|
| **Case 自动生成** | 中 | 给定 Prompt，LLM 生成测试用例 + 预期输出 |
| **对抗性生成** | 中 | 红队测试模式，专门构造让模型失败的输入 |
| **多维能力评估** | 高 | 把单一的 Deterministic/Rubric 拆成 6-8 个维度，每个维度独立的评分逻辑 |
| **失败聚类** | 中 | 对失败的 cases 做语义聚类，自动命名失败模式 |
| **根因分析** | 中 | LLM 分析失败模式的根因（Prompt 问题 vs 模型问题） |
| **Baseline 管理** | 低 | 每个 Asset 标记一个 Baseline 版本，所有 eval 自动与之对比 |
| **自动回归触发** | 低 | Commit 后 Webhook 触发 eval（或轮询检测文件变更） |
| **在线监控采样** | 高 | 生产环境调用采样存储，输出分布分析，Drift Detection |
| **用户反馈收集** | 低 | 👍/👎 接口，与 eval 结果关联 |

## 五、实施路线图

### Phase A：建立多维评估（4-6 周）
1. 后端：实现 6 维能力评估框架（指令遵循、事实准确、上下文理解、安全合规、表达质量、创造性）
2. 前端：能力雷达图 + 雷达图历史动画
3. 后端：Baseline 版本管理
4. 前端：VersionTreeView 升级，显示 vs Baseline 变化

### Phase B：AI 驱动诊断（4-6 周）
1. 后端：失败聚类 + 根因分析
2. 前端：诊断工作台（三栏布局）
3. 后端：自动修复建议生成
4. 前端：建议 Diff + 一键应用

### Phase C：持续评估（6-8 周）
1. 后端：自动回归触发（Commit → Eval）
2. 前端：CI 风格的时间线界面
3. 后端：在线监控采样 + Drift Detection
4. 前端：生产监控 Dashboard

### Phase D：Coverage 体系（4-6 周）
1. 后端：Case 自动生成 + 对抗性生成
2. 前端：Coverage Studio（雷达图 + 盲区高亮 + Case 矩阵）
3. 后端：覆盖度指标计算

---

## 六、一句话总结

> **Eval 1.0 是"工具"，Eval 2.0 是"体系"。**
>
> 1.0 回答"这个 Prompt 得了几分"；
> 2.0 回答"这个 Prompt 在什么能力上强/弱，为什么，怎么改，改了之后在线表现如何"。
